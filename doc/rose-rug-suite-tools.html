Rose User Guide: Suite Tools

Introduction

This chapter of the user guide discusses the Rose suite tools and utilities that can be used to simplify the logic of a suite and/or provide a common approach of doing the same things.

General Usage

The tools and utilities discussed in this chapter are accessed via Unix/Linux commands that have a common syntax:

rose UTIL [OPTS] [ARG ...]

See Rose Reference Guide: CLI for a complete reference.

All commands described in this chapter accept the following standard options:

--debug
This option is useful for debugging purpose as it would instruct the system to print a traceback on error.
--quiet, -q
Each --quiet or -q on the command line decrements the verbosity of the output by 1. The default verbosity is 1. If you want to command to only print what is essential to the STDOUT and suppress warnings on STDERR, you can specify a single --quiet to reduce the verbosity to 0.
--verbose, -v
Each --verbose or -v on the command line increments the verbosity of the output by 1. The default verbosity is 1. The highest verbosity is 3. A single --verbose would increase the verbosity to STDOUT. A double --verbose --verbose or -v -v would increase the verbosity to the highest level. (More --verbose can be specified on the command line, but would probably have no effect.)

rose suite-run

Some aspects of the rose suite-run command has already been discussed in A Brief Tour. Here, we are going to discuss some other features of the command.

See Rose Reference Guide: CLI > rose suite-run for a full command reference.

See Rose Reference Guide: Configuration > Suite Configuration for more information on how to set up a Rose suite configuration.

Optional Configuration Selection

If a suite configuration is used in different ways, each requiring a set of coordinated changes, the optional configuration may come in handy. Optional configuration files are stored in the opt/ directory of a suite configuration. Each file has a name that looks like opt/rose-suite-KEY.conf. One or more optional configurations can be selected using the --opt-conf-key=KEY or -O KEY option of the rose suite-run command.

For each specified optional configuration, the content of its file is appended to the main configuration file before the whole configuration is parsed. This allows settings to accumulate and/or be overridden in the correct order.

Re-define Configuration

If all you need to do is to test out a small change to a suite configuration, it may be easier to use the --define=[SECTION]KEY=VALUE or -D [SECTION]KEY=VALUE option on the command line. This option overrides everything in the main and optional configurations. E.g.:

# Set [env]FOO=foo, and [env]BAR=bar
# (Overridding any original settings of [env]FOO or [env]BAR)
rose task-run -D '[env]FOO=foo' -D '[env]BAR=bar'

# Switch off [env]BAZ
rose task-run -D '[env]!BAZ='

rose task-run

The command uses the suite engine environment to set up some standard ROSE_* environment variables, site/user specific PATH and similar variables, etc. for the task before launching an application or a task utility.

See Rose Reference Guide: CLI > rose task-run for a full command reference.

See Rose Reference Guide: Configuration > Application Configuration for more information on how to set up a Rose application configuration.

Application Configuration Selection

In a normal case, the command will look at the suite's app/ directory for an application configuration that matches the name of the task, and run the application based on the configuration. If you need to select an alternate application configuration or if the name of the task would not match any application configuration of the suite, you can use the --app-key=KEY option or the ROSE_TASK_APP environment variable to specify the name of the application configuration the task should use.

Command Selection

The application runner uses the following logic to select a shell command to run:

  1. If arguments are supplied to rose task-run, the arguments are run as a shell command.
  2. If the --command-key=KEY option is specified, the command specified in the [command]KEY setting in the application configuration file is used.
  3. If the [command] section in the application configuration file has a key matching the task name, the command specified in this setting is used.
  4. Otherwise, the command specified in the [command]default setting is used.

This mechanism allows, for example, similar tasks to share the same application configuration, if most of their differences can be defined at the command line.

Optional Configuration Selection

If an application configuration is shared by a set of tasks, each requiring a set of coordinated changes, the optional configurations may come in handy. Optional configuration files are stored in the opt/ directory of an application configuration. Each file has a name that looks like opt/rose-app-KEY.conf. One or more optional configurations can be selected using the --opt-conf-key=KEY or -O KEY option of the rose task-run command.

For each specified optional configuration, the content of its file is appended to the main configuration file before the whole configuration is parsed. This allows settings to accumulate and/or be overridden in the correct order.

Re-define Configuration

If all you need to do is to test out a small change to an application configuration, it may be easier to use the --define=[SECTION]KEY=VALUE or -D [SECTION]KEY=VALUE option on the command line. This option overrides everything in the main and optional configurations. E.g.:

# Set [env]FOO=foo, and [env]BAR=bar
# (Overridding any original settings of [env]FOO or [env]BAR)
rose task-run -D '[env]FOO=foo' -D '[env]BAR=bar'

# Switch off [env]BAZ

Task Utility Selection

Apart from running applications normally, the command also has a set of built-in utilities to call. You can use the --util-key=KEY option or the ROSE_TASK_UTIL environment variable to specify a task utility to run instead of an application.

If neither --no-auto-util or --util-key=KEY is specified, the task name will be passed to each built-in task utility for a pattern match. If there is a match, it will run the task using the task utility instead of as a Rose application.

A task utility would normally behave very much like running a Rose application. (Consider the normal application runner as the default task utility.) The key differences are normally that:

  1. A task utility may use a different working directory to a Rose application.
  2. A task utility will run some pre-defined commands or logics instead of a command defined in the application configuration.

Task Utility: fcm_make

A pair of built-in task utilities are provided for running fcm make.

The fcm_make utility will be invoked automatically if a task has a name that starts with fcm_make*. (However, if the task name starts with fcm_make2* it will invoke the fcm_make2 utility instead, see next paragraph).

The fcm_make2 utility will be invoked automatically if a task has a name that starts with fcm_make2*. If there is a Rose application configuration for its corresponding fcm_make* task, it will use its settings by default.

The fcm_make task utility expects a file file/fcm-make.cfg in its application configuration. It runs fcm make using this configuration file.

The fcm make task utility will also look for a matching fcm_make2* task in the suite which would normally be a continuation of the fcm make command as a separate USER@HOST. If so, will export an environment variable ROSE_TASK_MIRROR_TARGET to the fcm make command to substitute the mirror target.

You can configure these task utilities with environment variables or settings in rose-app.conf. (Settings in rose-app.conf override their equivalent environment variables.)

use-pwd=true: By default, the fcm_make and fcm_make2 task utilities change the working directory to $ROSE_SUITE_DIR/share/$ROSE_TASK_NAME. This option will stop this, and the working directory is the normal working directory of the task.

opt.jobs=N or the environment variable ROSE_TASK_N_JOBS: This can be used to control the number of processes fcm make would use in parallel. (default=4)

opts or the environment variable ROSE_TASK_OPTIONS: This can be used to pass extra options to the fcm make command.

Task Utility: rose_install

This built-in utility behaves almost the same as the application launcher. It will export the environment and install files, but will not launch a command.

In automatic task utility selection mode, this utility will be invoked automatically if a task has a name that starts with rose_install*.

By default, as for the application launcher, this task utility will use the following logic to determine the root directory to install file targets with a relative path:

  1. If the setting file-install-root=PATH is specified (at the root level) in the application configuration file, its value will be used.
  2. If the environment variable ROSE_FILE_INSTALL_ROOT is specified, its value will be used.
  3. Otherwise, the working directory of the task will be used.

rose task-env

There are times when extra environment needs to be defined before launching rose task-run. This is where rose task-env may come in handy. The command prints to the STDOUT the standard Rose task environment variables (which are normally provided by rose task-run) in a syntax compatible to bash / ksh. This means that the output of this command can be shell eval into the current environment. E.g.

eval $(rose task-env)

See Rose Reference Guide: CLI > rose task-env for a full list of environment variables provided by this command.

rose suite-hook

This command provide a common way to handle suite and task events of a Cylc suite. The command is normally called as a handler in the event hooks section of a Cylc suite's suite.rc file. For a task event for a remote task, it would attempt to pull the task logs back from the remote host running the task. For all other events, it will update the suite log view. If the --mail option is specified on the command line, the command will email you with the message of the event. This may be useful for monitoring failures or timeouts. If the --shutdown option is specified, the command will attempt to shutdown the suite immediately. This may be useful, for example, if you want the suite to shutdown immediately on any failures.

See Rose Reference Guide: CLI > rose suite-hook for a full list of environment variables provided by this command.

rose suite-log-view

This command inspects the log of a running or stopped suite to generate a view and display it by launching a web browser (which must be compliant to XHTML 1.0, JavaScript 1.8 and CSS 2.1). The rose suite-run command calls (the library of) this command to generate the initial view. The rose suite-hook command calls (the library of) this command to update the view on task events. If you are already in the directory of a suite, you can use this command to bring up its output on a web browser. Or if for whatever reasons rose suite-hook has failed to update the view, you can use this command to force an update.

See Rose Reference Guide: CLI > rose suite-log-view for a full list of environment variables provided by this command.

Other Useful Tools

rose suite-gcontrol

A convenient way to launch cylc gui. In the absence of arguments, the command assumes that base name of the current directory is the name of the suite you want to launch cylc gui on. If your suite is running on a dedicated suite host as specified by the Rose site/user configuration, it will automatically connect to the suite at the correct host.

See Rose Reference Guide: CLI > rose suite-gcontrol for a full list of environment variables provided by this command.

rose suite-scan

A wrapper of cylc scan. The command scans for your running suites in the dedicated suite hosts as specified by the Rose site/user configuration.

See Rose Reference Guide: CLI > rose suite-scan for a full list of environment variables provided by this command.

rose app-run

Run an application according to its configuration, outside of a suite task environment. Although you will normally launch a Rose application using rose task-run, there are situations when you may have a standalone Rose application configuration that you just want to run outside of a suite. This is where rose app-run may come in handy.

See Rose Reference Guide: CLI > rose app-run for a full list of environment variables provided by this command.
